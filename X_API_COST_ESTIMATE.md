# X API従量課金モデル：コスト試算（月6,604 Posts使用想定）

## 📊 **料金体系（2026年2月〜）**

### 従量課金の単価

| 項目 | 単価 |
|------|------|
| **Post（読み取り）** | $0.005 / Post |
| **User（ユーザー情報取得）** | $0.01 / User |
| **Post（作成）** | $0.01 / Post |
| **DM Event（読み取り）** | $0.01 / DM Event |
| **DM Interaction（作成）** | $0.01 / DM |

**重要な仕組み：24時間重複排除**
- 同じPostを24時間以内に複数回取得しても**1回のみ課金**
- 毎日UTC 0:00にリセット
- すべてのリソース（Posts、Usersなど）に適用

## 💰 **月間コスト試算（15,000 Posts使用）** ⚠️ 修正版

### 前提条件
- **月間Post読み取り**: 15,000 Posts（ほぼ上限まで使用）
- **評価対象学生数**: 約150名（推定）
- **評価頻度**: 月2回
- **1名あたりAPIリクエスト**: 
  - ユーザー情報取得: 1回
  - ツイート取得: 1〜2回（平均100 Posts/名）

### コスト計算

#### **1. Post読み取りコスト** ⚠️
```
15,000 Posts × $0.005 = $75.00
```

#### **2. User情報取得コスト（推定）**
このシステムは各学生のユーザー情報を取得します：
- 評価対象: 150名
- 評価頻度: 月2回
- User情報取得: 150名 × 2回 = 300リクエスト

**ただし24時間重複排除が適用される可能性：**
- 同日にバッチ評価を2回実行 → 2回目は課金なし
- 異なる日に実行 → 各回課金

**最大コスト（重複排除なし）:**
```
300 User lookups × $0.01 = $3.00
```

**最小コスト（同日実行で重複排除）:**
```
150 User lookups × $0.01 = $1.50
```

#### **3. 総コスト**

| シナリオ | Post読み取り | User取得 | **月間合計** |
|---------|------------|---------|-------------|
| **最大コスト** | $75.00 | $3.00 | **$78.00** |
| **最小コスト** | $75.00 | $1.50 | **$76.50** |

**推定月間コスト: $76.50〜$78.00**

## 🎯 **コスト最適化の施策** ⚠️ 重要性UP

### 現在の実装（すでに対策済み）

#### ✅ 1. キャッシュ機能（最重要）
- 評価結果をスプレッドシートに保存
- 再評価時はAPIを呼ばずキャッシュから取得
- **コード**: `src/lib/evaluation-cache.ts`

**効果（月15,000 Posts使用時）:**
- キャッシュヒット率50%と仮定: 月間コスト**$75 → $37.50**（半減）
- **年間節約額: $450**

**⚠️ 現在の使用量では最も効果的な最適化**

#### ✅ 2. 24時間重複排除の活用
- 同日にバッチ評価を複数回実行してもUser情報は1回のみ課金
- Posts読み取りも同一Post IDなら重複排除

**効果:**
- User情報取得: $3 → $1.50（半減）
- **年間節約額: $18**

#### ✅ 3. レート制限検出
- 429エラーを検出して処理をスキップ
- `rateLimited: true` フラグを返す

**効果:**
- 無駄なリトライによるコスト増を防止

### 追加で実装可能な最適化（優先度順）

#### 🔴 4. キャッシュ有効期限の延長（最優先）
**現在の問題:**
- キャッシュの有効期限が不明確
- 毎回APIリクエストしている可能性

**最適化案:**
```typescript
// キャッシュが30日以内なら再利用
const cacheAge = Date.now() - cachedData.timestamp;
const CACHE_VALID_PERIOD = 30 * 24 * 60 * 60 * 1000; // 30日

if (cacheAge < CACHE_VALID_PERIOD) {
  return cachedData; // APIリクエストなし
}
```

**効果:**
- 月2回評価の場合、2回目は全員キャッシュヒット
- 月間コスト: $75 → **$37.50**（半減）
- **年間節約額: $450**

#### 🟡 5. バッチ評価のスケジューリング最適化
**現在の問題:**
- 異なる日に評価 → User情報が重複課金

**最適化案:**
- 月2回の評価を**同日の午前・午後**に実行
- User情報の重複排除が適用され、2回目は$0

**効果:**
- User情報取得コスト: $3 → $1.50（$1.50削減）
- **年間節約額: $18**

#### 🟡 6. ツイート取得数の制限
**現在の問題:**
- 1名あたり平均100 Posts取得（15,000 Posts ÷ 150名）
- 月初〜月末のツイートを全取得している可能性

**最適化案:**
```typescript
// ツイート取得数を制限
const maxResults = 50; // 100 → 50に削減
await fetchRecentTweets(bearerToken, userId, maxResults, targetMonth);
```

**効果:**
- Posts使用量: 15,000 → 7,500（半減）
- 月間コスト: $75 → **$37.50**
- **年間節約額: $450**

**⚠️ 注意:**
- 評価精度が低下する可能性
- 月に50ツイート以上投稿する学生のデータが不完全になる

#### 🟢 7. X評価の選択的実施
**最適化案:**
- Xアカウントが活発な学生のみ評価（月10ツイート以上など）
- 非活発アカウントは前月のキャッシュを使用

**効果:**
- 評価対象を50%削減した場合: $75 → **$37.50**
- **年間節約額: $450**
