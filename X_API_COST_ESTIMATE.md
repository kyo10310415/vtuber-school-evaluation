# X API従量課金モデル：コスト試算（月6,604 Posts使用想定）

## 📊 **料金体系（2026年2月〜）**

### 従量課金の単価

| 項目 | 単価 |
|------|------|
| **Post（読み取り）** | $0.005 / Post |
| **User（ユーザー情報取得）** | $0.01 / User |
| **Post（作成）** | $0.01 / Post |
| **DM Event（読み取り）** | $0.01 / DM Event |
| **DM Interaction（作成）** | $0.01 / DM |

**重要な仕組み：24時間重複排除**
- 同じPostを24時間以内に複数回取得しても**1回のみ課金**
- 毎日UTC 0:00にリセット
- すべてのリソース（Posts、Usersなど）に適用

---

## 💰 **月間コスト試算（6,604 Posts使用）**

### 前提条件
- **月間Post読み取り**: 6,604 Posts
- **評価対象学生数**: 約100名（推定）
- **評価頻度**: 月2回
- **1名あたりAPIリクエスト**: 
  - ユーザー情報取得: 1回
  - ツイート取得: 1〜2回（平均66 Posts/名）

### コスト計算

#### **1. Post読み取りコスト**
```
6,604 Posts × $0.005 = $33.02
```

#### **2. User情報取得コスト（推定）**
このシステムは各学生のユーザー情報を取得します：
- 評価対象: 100名
- 評価頻度: 月2回
- User情報取得: 100名 × 2回 = 200リクエスト

**ただし24時間重複排除が適用される可能性：**
- 同日にバッチ評価を2回実行 → 2回目は課金なし
- 異なる日に実行 → 各回課金

**最大コスト（重複排除なし）:**
```
200 User lookups × $0.01 = $2.00
```

**最小コスト（同日実行で重複排除）:**
```
100 User lookups × $0.01 = $1.00
```

#### **3. 総コスト**

| シナリオ | Post読み取り | User取得 | **月間合計** |
|---------|------------|---------|-------------|
| **最大コスト** | $33.02 | $2.00 | **$35.02** |
| **最小コスト** | $33.02 | $1.00 | **$34.02** |

**推定月間コスト: $34〜$35**

## 🎯 **コスト最適化の施策**

### 現在の実装（すでに対策済み）

#### ✅ 1. キャッシュ機能
- 評価結果をスプレッドシートに保存
- 再評価時はAPIを呼ばずキャッシュから取得
- **コード**: `src/lib/evaluation-cache.ts`

**効果:**
- 2回目以降の評価: $0（APIリクエストなし）
- キャッシュヒット率50%と仮定: 月間コスト**$17〜$18**（半減）

#### ✅ 2. 24時間重複排除の活用
- 同日にバッチ評価を複数回実行してもUser情報は1回のみ課金
- Posts読み取りも同一Post IDなら重複排除

**効果:**
- User情報取得: $2 → $1（半減）

#### ✅ 3. レート制限検出
- 429エラーを検出して処理をスキップ
- `rateLimited: true` フラグを返す

**効果:**
- 無駄なリトライによるコスト増を防止

### 追加で実装可能な最適化

#### 🟡 4. バッチ評価のスケジューリング最適化
**現在の問題:**
- 異なる日に評価 → User情報が重複課金

**最適化案:**
- 月2回の評価を**同日の午前・午後**に実行
- User情報の重複排除が適用され、2回目は$0

**効果:**
- User情報取得コスト: $2 → $1（$1削減）

#### 🟡 5. APIリクエストの優先度付け
**最適化案:**
- X評価が必須の学生のみAPIリクエスト
- Xアカウント未設定の学生はスキップ（すでに実装済み）
- キャッシュが1ヶ月以内の場合は再取得しない

**効果:**
- リクエスト数を20〜30%削減可能
